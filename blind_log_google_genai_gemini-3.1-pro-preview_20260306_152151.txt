Imagine a user has the seed phrase: **`zebra patient timber unaware modify scatter right filter suspect useless involve typical`**

When the user types this phrase into a new wallet, the software does not simply store the words. Instead, it executes a series of cryptographic functions defined by Bitcoin Improvement Proposals (BIPs) 39, 32, and 44/84. Here is exactly what happens under the hood:

### Step 1: Mnemonic to Binary Seed (BIP39)
First, the wallet software converts the 12 words back into a raw binary seed. It does this using a key stretching function called **PBKDF2** (Password-Based Key Derivation Function 2).
*   **The Input:** The 12 words act as the "password."
*   **The Salt:** The string "mnemonic" plus an optional user-defined passphrase (if none is used, it's just "mnemonic").
*   **The Process:** The software runs the input and salt through the HMAC-SHA512 hashing algorithm **2,048 times**. 
*   **The Output:** This produces a 512-bit (64-byte) raw binary seed.

### Step 2: Generating the Master Key (BIP32)
Now that the wallet has the 512-bit seed, it needs to create the root of the wallet, known as the Master Node.
*   The 512-bit seed is split in half.
*   The left 256 bits become the **Master Private Key**.
*   The right 256 bits become the **Master Chain Code** (which introduces entropy for deriving child keys).
Together, these form the Extended Master Private Key (often starting with `xprv`).

### Step 3: Deriving Child Keys (BIP44 / BIP84)
A modern Bitcoin wallet doesn't just use one address; it generates a new one for every transaction to protect privacy. It uses a hierarchical deterministic (HD) tree structure to do this. The wallet follows a specific "Derivation Path" to find the right keys. 

For a modern Native SegWit Bitcoin wallet (BIP84), the path looks like this: `m/84'/0'/0'/0/0`
*   `m`: Master node
*   `84'`: Purpose (BIP84 - Native SegWit)
*   `0'`: Coin type (0 is Bitcoin)
*   `0'`: Account number (Account 0)
*   `0`: Change (0 for receiving addresses, 1 for change addresses)
*   `0`: Address index (The very first receiving address)

The wallet uses the Master Private Key, the Master Chain Code, and the derivation path to mathematically calculate the specific private key for that exact address index.

### Step 4: Private Key to Public Key
Once the wallet has derived the specific 256-bit private key for an address, it uses **Elliptic Curve Cryptography (secp256k1)** to generate the corresponding Public Key. This is a one-way mathematical function; you can easily calculate the public key from the private key, but it is practically impossible to reverse the math to find the private key from the public key.

### Step 5: Public Key to Bitcoin Address
Finally, the wallet converts the Public Key into a readable Bitcoin address that the user can share to receive funds.
*   The public key is hashed using **SHA-256**.
*   That result is hashed again using **RIPEMD-160** (this shortens it and adds a layer of security).
*   The result is encoded into the final address format. For modern wallets, this uses **Bech32** encoding, resulting in an address that starts with `bc1q...`

### The Recovery Conclusion
The wallet software repeats Steps 3 through 5 rapidly, checking the blockchain for any history associated with addresses at index 0, 1, 2, 3, and so on. Once it finds the user's past transactions, it syncs the balances, and the wallet is fully recovered—all originating from that initial 12-word phrase.
